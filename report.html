Position Engine Performance Report

Local smoke-test results from the case-study implementation.

Summary

The local tests separate the pure in-memory position engine from the SQLite-backed durable ingestion path.

The core position calculation is fast. The local bottleneck is durable persistence, specifically per-event SQLite transaction/SQL overhead. This is expected because SQLite is used for local correctness and recovery, not as the proposed production throughput store.

Measured Results

SQLite-backed service ~1,450 events/sec
Engine only ~84,065 events/sec
Read latency ~0.002 ms
Mode Events Elapsed Throughput Average Read Latency

Requirement Interpretation

Requirement Local Result / Design Answer
Baseline throughput: 1,000 events/sec Met locally by the SQLite-backed service path at approximately 1,450 events/sec in this smoke test.
Peak burst: 10,000 events/sec The in-memory engine exceeds this level, but the local SQLite durable path does not. Production scaling would use a production durable store and sharded workers by tradeId.
Read latency under 5ms Met locally; reads from the in-memory read model are far below 5ms.
Zero-data-loss recovery Events are written durably before being applied to memory. On restart, the app replays the durable event log to rebuild exact state.

Profile Findings

The most expensive operations were in the SQLite persistence path:

Operation Cumulative Time Meaning
app/store.py append_event 6.639 sec Durable event insert path, including duplicate check, insert, and transaction handling.
sqlite3.Connection.__exit__ 3.312 sec Transaction commit/cleanup for each stored event.
sqlite3.Connection.execute 3.140 sec SQL statements for duplicate lookup and event insert.
app/engine.py apply_event 0.197 sec Actual position state transition.

Production Scaling Conclusion

The correct production optimization is not to hold accepted events in memory and batch them unsafely. The correct strategy is:

durably accept events first
process durable events by tradeId shard
keep hot reads in a shared cache/read model
use checkpoints to bound recovery time

This preserves the finance-critical rule that accepted events must be recoverable, while still allowing horizontal scaling for burst traffic.
